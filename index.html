The Problem

Finance data is never as clean as the dashboard

Transaction records, billing exports, and FX rates arrive on different schedules, in different currencies, with different keys. A handful of duplicate transaction IDs or a late-arriving FX rate is enough to make two systems disagree about revenue, and someone has to find out why before it becomes a reporting problem.

This project simulates that exact situation using three real-world-shaped datasets, deliberately seeded with the failure modes a Finance data engineer actually deals with: mismatched currencies, duplicate keys, late-arriving records, and inconsistent date formats.

Architecture

Four stages, each with its own validation gate

Built in PySpark end to end. Every stage writes to its own table and runs a data quality check before the next stage is allowed to read from it.

01 — Ingest

Multi-source load

Reads transactions, billing, and FX data with explicit schemas, handling three different date formats and two currency conventions on the way in.

02 — Validate

Row-level checks

Null checks, duplicate-key detection via window functions, and referential checks confirming every transaction has a matching FX rate.

03 — Reconcile

Cross-source join

Normalizes every transaction to USD using the correct date's FX rate, then joins against billing to surface mismatches to the cent.

04 — Output

Analyst-ready table

A single reconciled table with a discrepancy flag and reason code, built to be queried directly by a Finance analyst, no notebook required.

reconcile_revenue.sql — duplicate detection with a window function
-- Flag transactions that share an ID but disagree on amount or currency WITH ranked AS ( SELECT *, ROW_NUMBER() OVER ( PARTITION BY transaction_id ORDER BY ingested_at DESC ) AS rn FROM transactions_raw ) SELECT transaction_id, amount, currency, ingested_at FROM ranked WHERE rn = 1;
Ownership

Data quality, end to end

Every stage fails loudly and specifically. No silent row drops, and no generic stack traces standing in for an actual explanation.

Automated checks per stage

  • Row count deltas between raw and validated tables, with a tolerance threshold
  • Null checks on required join keys before any reconciliation runs
  • Referential integrity: every transaction date has a matching FX rate
  • Revenue total tolerance check between reconciled output and source billing totals

What "failing loudly" means here

  • Each check raises a named exception with the offending row count and a sample of failing keys
  • Pipeline halts before writing a partially-reconciled table downstream
  • Logs are written per-stage, so a failure at reconciliation doesn't require re-reading ingestion logs to diagnose
Incident log — real issue hit while building this

Symptom: reconciled revenue was off by ~0.4% for one week of data only.
Root cause: FX rates table had a gap for a single holiday date; the join defaulted those rows to a null rate, which the currency conversion silently coerced to zero instead of failing.
Fix: added a referential integrity check before reconciliation that halts the pipeline if any transaction date is missing a matching FX rate, instead of allowing a null to pass through.
